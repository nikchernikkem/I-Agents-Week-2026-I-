## LLM без инструментов

LLM без доступа к инструментам:
1. Имеют ограниченные возможности из-за специфики обучения
2. Не имеют доступа к актуальной информации
3. Не могут самостоятельно выполнять действия

---

## Что такое инструменты

**Tool (инструмент)** — функция, которую LLM может вызвать для взаимодействия с внешними системами.

Инструмент должен включать:
- **Описание** — что он делает (текстовое)
- **Исполняемую функцию** — реализация в рантайме
- **Входные аргументы** — с типами
- **Выход** — с типом

---

## Как работает вызов инструментов

LLM только генерируют текст — они **не исполняют код**. Рантайм агента перехватывает вызов инструмента из вывода модели, выполняет его и возвращает результат обратно модели.

### Как предоставить инструменты LLM

- Модель **информируется** о доступных инструментах через system prompt
- Модель **обучена** генерировать вызовы инструментов (специальные токены, формат)
- Описания инструментов должны быть **ясными, точными и непротиворечивыми**

---

## Принципы проектирования инструментов

| Принцип | Описание |
|---------|----------|
| **Обработка частичных сбоев** | Инструменты должны обрабатывать частичные сбои, сохраняя консистентность системы |
| **Идемпотентность** | Повторные вызовы (после таймаутов, рестартов) не должны дублировать побочные эффекты |
| **Понятные ошибки** | Ошибки должны быть понятны LLM для принятия решения: запросить параметры, повторить, предложить альтернативу |
| **Подсказки следующих шагов** | Инструмент может возвращать guidance для продолжения диалога (запросить дату, формат телефона) |
| **Структурированный вывод** | Возвращать машиночитаемые структуры вместо свободного текста: `{status, order_id, delivery_window}` |

> Хороший инструмент для AI-агента — это не просто функция. Это надёжный контракт между LLM и внешними системами, спроектированный для устойчивости к сбоям, повторам и изменяющейся среде.

## Когда нужен протокол

Инструменты значительно расширили возможности LLM, но:
- Инструменты встроены внутрь каждого агента
- API меняются со временем
- Разные модели ожидают разные форматы вызовов
- Переиспользование инструментов остаётся низким
- Изменение API требует обновления множества агентов
- Системы становятся хрупкими и плохо масштабируются

---

## Model Context Protocol (MCP)

> **MCP** — открытый стандарт, обеспечивающий безопасные двусторонние соединения между AI-моделями и внешними системами (API, инструменты, источники данных).

Опубликован Anthropic 25 ноября 2024 года.

**MCP — это «USB-C порт» или «HTTP-слой» экосистемы AI.**

- [Анонс Anthropic](https://www.anthropic.com/news/model-context-protocol)
- [Документация MCP](https://modelcontextprotocol.io/docs/getting-started/intro)

---

## Архитектура и роли MCP

Клиент-серверная архитектура с тремя ролями:

### Host (Хост)

- Оркестрирует LLM и AI-воркфлоу
- Управляет соединениями: один MCP client на MCP server
- Контролирует UI (Cursor, кастомные агенты)
- Безопасность и разрешения: auth, лимиты, политики
- Согласие пользователя: одобрение на доступ к данным и инструментам

### Server (Сервер)

- Провайдер контекста и capabilities
- Лёгкая программа, экспонирующая инструменты, контекст и возможности
- Может работать локально или удалённо
- Регистрирует capabilities, обрабатывает запросы, предоставляет контекст, отправляет обновления

### Client (Клиент)

- Коннектор между Host и Server
- Протокол: JSON-RPC 2.0
- Согласование capabilities: фичи и версии при инициализации
- Исполнение инструментов: отправляет вызовы, возвращает результаты
- Обработка обновлений от сервера в реальном времени

## Какие проблемы решает MCP

| Проблема | Без MCP | С MCP |
|----------|---------|-------|
| **Переиспользование инструментов** | Каждая команда реализует одни и те же инструменты | Инструменты живут в MCP-серверах как standalone-компоненты. Написал один раз — используй везде |
| **Обновление API** | Все команды агентов обновляют интеграции вручную | Ответственность переходит к maintainer-ам MCP-сервера. Агенты автоматически обнаруживают обновлённые capabilities |
| **Масштабирование и модификации** | Компоненты тесно связаны; изменение одного требует переписывания многих | Разделение ответственности. Изменения локализованы в отдельных компонентах |

---

## О переусложнении

> **Do not over-engineer. Start simple. Add MCP when complexity forces you.**

В промежуточном варианте можно использовать лёгкие экосистемы инструментов (например, [AgentSkills](https://agentskills.io/home)):
- Готовые навыки
- Стандартизированные интерфейсы
- Быстрая разработка без тяжёлой инфраструктуры
- Ступенька к более структурированным архитектурам вроде MCP
