## Как думает агент: TAO-цикл

Базовая схема работы агента — **TAO-цикл (TAO loop)**:

- **Мысль (Thought)**
- **Действие (Action)**
- **Наблюдение (Observation)**

TAO — это workflow AI-агента, который позволяет динамически корректировать ответы, улучшать понимание задачи и повышать точность результата.

### Общая схема
1. **Запрос пользователя**
2. **Thought** — модель рассуждает и строит план
3. **Action** — выполняется действие или вызов инструмента (tool call)
4. **Observation** — агент получает результат из среды
5. Если задача не решена — цикл повторяется
6. Если решена — возвращается **финальный ответ**

### Зачем нужен TAO-цикл
TAO делает агента:
- итеративным;
- чувствительным к обратной связи;
- способным исправлять собственные ошибки;
- пригодным для работы с внешними инструментами.

## Этап Thought: внутреннее рассуждение

На этапе **Thought** агент не действует, а **думает**: анализирует вход, строит план и определяет следующий шаг.

### Типы рассуждения

| Тип | Пример |
| --- | --- |
| **Планирование (Planning)** | «Я разобью задачу на 3 шага: найти бронирование → проверить правила → обновить» |
| **Анализ (Analysis)** | «Судя по ошибке API, проблема в формате даты» |
| **Саморефлексия (Self-reflection)** | «Мой предыдущий ответ был слишком общим» |

Такие типы мышления показывают, что Thought — это не просто «подумать перед ответом», а отдельная полезная рабочая фаза.

### Важный нюанс
В моделях, обученных на **вызове функций**, фаза Thought может быть **неявной**: модель сразу генерирует вызов функции без явного текстового рассуждения. Однако в агентных системах явное рассуждение обычно улучшает качество, и этот этап можно специально сохранять в дизайне агента.

## Стратегии рассуждения: цепочка рассуждений (Chain of Thought, CoT), ReAct, reasoning

В лекции выделяются три основные стратегии.

### 1. Цепочка рассуждений (Chain of Thought, CoT)
Пошаговое рассуждение внутри одной генерации.

**Особенности:**
- рассуждение шаг за шагом;
- без инструментов;
- одна генерация;
- хорошо подходит для логики и математики.

**Риск:** если ошибка возникла на одном шаге, все дальнейшее рассуждение строится на неверной основе.

### 2. ReAct
Подход, где чередуются:
- **Думай (Think)**
- **Действуй (Act)**
- **Наблюдай (Observe)**

Подходит для задач с инструментами и внешней средой. После каждого действия агент получает наблюдение (Observation) и может скорректировать план.

### 3. Рассуждающая модель (reasoning)
Это не просто стратегия промптинга, а свойство модели, встроенное на этапе обучения. В лекции приводятся модели типа **OpenAI o1** и **DeepSeek R1**.

**Особенность:** модель сама решает, сколько итераций внутреннего рассуждения ей нужно и в каком формате они будут происходить. Для разработчика это в значительной степени «черный ящик».

---

### Сравнение стратегий

| Характеристика | Цепочка рассуждений (CoT) | ReAct | Рассуждающая модель (Reasoning) |
|---|---|---|---|
| Пошаговая логика | Да | Да | Да |
| Внешние инструменты | Нет | Да | Опционально |
| Коррекция плана | Нет | Да, через наблюдение (Observation) | Внутренняя |
| Где задается | Промптом | Промптом | На этапе обучения |
| Лучше всего подходит | Логика, математика | Задачи с инструментами | Сложное планирование |
| Стоимость | Низкая | Средняя | Высокая |

### Практический вывод
Для задач с инструментами **ReAct лучше CoT**.

### Практическое дерево выбора
- Если нужны **инструменты**, в большинстве случаев подходит **ReAct**
- Если инструменты **не нужны**:
  - **CoT** — для более простых задач, когда важны скорость и дешевизна
  - **Reasoning** — когда нужна более глубокая многошаговая внутренняя проработка и CoT уже систематически не справляется

## Этап действия (Action): вызов инструментов

На этапе **действия** агент формирует конкретное действие.

Важно: на этом этапе агент **еще не исполняет** действие, а определяет, **какую функцию** и **с какими аргументами** нужно вызвать.

### Форматы вызова инструментов

| Формат | Описание | Пример |
|---|---|---|
| **JSON-агент** | Action как JSON-объект | `{"action": "search_flights", ...}` |
| **Вызов функций** | Нативный формат модели | `tool_calls` |
| **Кодовый агент** | Генерация исполняемого кода | `results = search_flights(...)` |

### Важное замечание о Code Agent
Кодовый агент генерирует исполняемый код, и это потенциально опасно. Такой код должен запускаться только в **изолированной среде** — например, в **Docker-контейнере** или **sandbox** — с явно ограниченными правами и доступами.

### Важный инженерный паттерн: остановиться и распарсить (Stop and Parse)

Правильный поток работы:

1. LLM генерирует action JSON
2. **STOP**
3. runtime парсит action и args
4. runtime вызывает tool
5. tool возвращает реальный результат

Это важно, чтобы модель не начинала «придумывать» результат инструмента вместо настоящего вызова.

### Важное уточнение
Если используется **function calling**, механизм остановки в значительной степени уже встроен в формат модели и дополнительно о нем обычно думать не нужно.

## Этап наблюдения (Observation): обратная связь от среды

**Наблюдение** — это результат действия агента.

### Три шага обработки
1. **Разбор** — определить функцию и аргументы
2. **Исполнение** — выполнить действие
3. **Добавление в контекст** — добавить результат в контекст агента

### Типы observation

| Тип | Пример |
| --- | --- |
| **Данные API** | `{"flights": [...], "count": 3}` |
| **Системная ошибка** | `{"error": "rate_limit_exceeded", "retry_after": 30}` |
| **Подтверждение** | `{"status": "booking_confirmed", "id": "BK-12345"}` |
| **Пользовательский ввод** | `"Yes, book this flight"` |

Observation — это то, что делает агента не «одним ответом», а полноценным циклическим процессом. Без шага наблюдения агент теряет значительную часть своей отличия от обычного промпта.

## Структурированный вывод для планирования

Структурированный вывод позволяет вместо свободного текста получать **формализованный план**.

Пример структуры:
- `SubTask`
  - `agent_name`
  - `description`
  - `priority`
- `Plan`
  - `subtasks: list[SubTask]`

Это позволяет строить агентные системы, в которых результат LLM сразу пригоден для дальнейшей машинной обработки.

### Практический смысл
Агент часто должен возвращать не просто текст, а строго заданный формат, который затем можно:
- использовать в коде;
- передавать другим агентам;
- валидировать автоматически.

## TAO-цикл на примере перебронирования рейса

### Сценарий

| Шаг | Мысль (Thought) | Действие (Action) | Наблюдение (Observation) |
| --- | --- | --- | --- |
| 1 | Найти бронирование | `get_booking(user_id="U-1234")` | `{booking_id: "BK-789", flight: "SU2454"}` |
| 2 | Найти рейсы на завтра | `search_flights("SVO", "CDG", "04-16")` | `[{SU2456: 37k₽}, {AF1147: 42k₽}]` |
| 3 | Проверить правила | `get_policy("rebooking")` | `{fee: 3000₽, same class only}` |
| 4 | Предложить варианты пользователю | `final_answer(...)` | Финальный ответ |

### Итог примера
- **4 итерации TAO-цикла**
- **3 вызова инструментов** (get_booking, search_flights, get_policy)
- **1 финальный ответ**

## Почему возникает потребность в мультиагентных системах (MAS)

Когда задача усложняется, один агент начинает:
- теряться при большом числе инструментов;
- путать контексты подзадач;
- становиться сложным для отладки;
- работать менее надежно.

### Практический ориентир
Из записи лекции следует важный практический сигнал: если инструментов становится **примерно больше 10–15**, одиночный агент начинает заметно путаться; при этом уже при **6+** стоит рассматривать мультиагентную архитектуру.

### Решение
Использовать **специализированных агентов**, где каждый отвечает за свою область знаний и свой участок работы.

## Одиночный агент vs мультиагентная система

| Критерий | Одиночный агент | Мультиагентная система |
|---|---|---|
| Сильные стороны | Простота, меньше координации, меньше ресурсов | Специализация, параллельность, отказоустойчивость |
| Слабые стороны | Ограничен контекстом, путается при большом числе инструментов | Накладные расходы на координацию (overhead), сложнее отладка |
| Когда выбирать | Простые задачи, 2–5 | Много инструментов, разнообразная экспертиза |

### Ключевые преимущества MAS
- более полное покрытие аспектов задачи;
- взаимная проверка и валидация;
- возможность использовать небольшие специализированные модели;
- устойчивость к сбоям отдельных агентов.

## 6 принципов мультиагентных систем

| Принцип | Смысл |
|---|---|
| **Декомпозиция (Decomposition)** | Разбиение задачи на подзадачи |
| **Специализация (Specialization)** | Каждый агент эксперт в своей области |
| **Коммуникация (Communication)** | Эффективный обмен информацией |
| **Координация (Coordination)** | Согласование действий и сбор результатов |
| **Верификация (Verification)** | Промежуточная проверка результатов |
| **Обратная связь (Feedback)** | Коррекция действий на основе ответов других |

## Типы взаимодействия агентов

### 1. Дебаты (Debate)
Агенты спорят между собой и приходят к более точному ответу.

**Когда полезно:**
- нужна высокая точность;
- фактчекинг;
- этика и риски.

**Минус:** медленно и дороже по токенам.

### 2. Коллаборация (Collaboration)
Разные агенты решают разные части задачи, а координатор агрегирует результат.

**Когда полезно:**
- понятный pipeline;
- разные навыки;
- эффективность.

**Минус:** нужно заранее хорошо продумать роли и структуру агентов.

### 3. Конкуренция (Competition)
Несколько агентов предлагают разные решения, а агент-оценщик (evaluator) выбирает лучшее.

**Когда полезно:**
- несколько стратегий;
- творческие задачи;
- важна вариативность решений.

**Минус:** самый дорогой по ресурсам подход, потому что несколько агентов независимо решают одну и ту же задачу целиком.

### 4. Передача (Handoff)
Ключевой collaboration-паттерн: один агент передает задачу другому вместе с нужным контекстом.

Пример:
- пользователь хочет перебронировать рейс и узнать про багаж;
- координатор делит задачу;
- **агент перебронирования (Rebook Agent)** получает handoff с данными о рейсе;
- **агент багажа (Baggage Agent)** получает handoff с правилами;
- затем координатор собирает итоговый ответ.

### Практический смысл handoff
Паттерн особенно полезен для задач с большим контекстом и многошаговой обработкой.

## Архитектуры мультиагентных систем

Рассматриваются 4 архитектуры:

1. **Иерархическая (Hierarchical, Vertical)**
2. **Децентрализованная (Decentralized, Horizontal)**
3. **Централизованная (Centralized, Star)**
4. **Общий пул сообщений (Shared Message Pool)**

### Сравнение архитектур

| Критерий | Иерархическая | Децентрализованная | Централизованная | Общий пул сообщений |
| --- | --- | --- | --- | --- |
| Сложность задач  | Высокая | Средне-высокая | Низко-средняя | Средняя |
| Отказоустойчивость | Средняя | Высокая | Низкая | Средняя |
| Масштабируемость | Средняя | Высокая | Низкая | Высокая |
| Простота отладки | Высокая | Низкая | Высокая | Средняя |
| Предсказуемость  | Высокая | Низкая | Высокая | Средняя |

### Короткая интерпретация
- **Иерархическая (Hierarchical)** — хорошо для сложных, но хорошо декомпозируемых задач;
- **Децентрализованная (Decentralized)** — для гибкости и отказоустойчивости;
- **Централизованная (Centralized)** — для простой маршрутизации;
- **Общий пул сообщений (Shared Pool)** — для асинхронных сценариев.

### Дополнительные замечания из лекции

#### Иерархическая архитектура
Плюсы:
- предсказуемость;
- легко отлаживать;
- хорошо подходит для сложных задач с понятной декомпозицией.

Минус:
- координатор становится единой точкой отказа и потенциальным «бутылочным горлышком».

Практическое замечание:
- именно эта архитектура часто используется в production.

#### Децентрализованная архитектура
Плюсы:
- самая высокая отказоустойчивость;
- хорошая масштабируемость;
- гибкая адаптация к разным ситуациям.

Минусы:
- сложная координация;
- меньше предсказуемости;
- труднее отлаживать;
- есть риск зацикливания, когда агенты бесконечно перекидывают задачи друг другу.

#### Централизованная архитектура
Ключевое отличие от иерархической:
- роутер не декомпозирует задачу, а только решает, **кому** ее отправить.

Плюсы:
- самая простая архитектура;
- легко отлаживать;
- высокая предсказуемость.

Минусы:
- единая точка отказа;
- плохо масштабируется;
- плохо подходит для сложных задач.

#### Общий пул сообщений (Shared Message Pool)
Принцип:
- агенты не общаются напрямую;
- они пишут сообщения в общий пул и читают из него;
- это похоже на **publisher-subscriber** подход.

Плюсы:
- высокая масштабируемость;
- независимая работа агентов;
- хорошо подходит для событийных сценариев.

Минусы:
- средняя сложность отладки;
- порядок обработки сообщений может быть непредсказуемым.

## Протоколы: MCP и A2A

| Протокол | Формат связи | Аналогия |
| --- | --- | --- |
| **MCP** | Агент ↔ Инструмент | USB-C для подключения AI к инструментам |
| **A2A** | Агент ↔ Агент | Wi‑Fi Direct для прямой связи устройств |

### Смысл
- **MCP** нужен для подключения агента к инструментам;
- **A2A** — для связи агентов между собой.

## Ограничения мультиагентных систем

| Категория | Проблема | Пример |
| --- | --- | --- |
| **Ресурсоемкость ** | Каждый агент = отдельный вызов LLM | 3 агента × 5 итераций = 15+ вызовов |
| **Накопление ошибок ** | Ошибка одного агента влияет на следующих | Один агент нашел не тот рейс, другой его забронировал |
| **Отладка ** | Трудно понять, где произошел сбой | Нужна трассировка каждого шага |

### Вывод
MAS дает гибкость и специализацию, но требует:
- контроля стоимости;
- трассировки;
- продуманной координации;
- инструментов наблюдаемости и логирования.

## Как выбирать архитектуру: гайд по принятию решений

### Логика выбора
1. **Нужен ли агент вообще?**  
   Если нет — используем простой API / обычную логику.

2. **Сколько инструментов?**
   - **2–5** → чаще достаточно одиночного агента
   - **6+** → стоит рассматривать мультиагентную систему (MAS)

3. **Нужна ли верификация?**
   - если да → **MAS + Critic**

4. **Как устроена задача?**
   - четкая иерархия → **Hierarchical**
   - равноправные эксперты → **Decentralized**
   - много подсистем → **Centralized**
   - асинхронная обработка → **Shared Pool**

### Главное правило
**Архитектуру выбирают под задачу, а не наоборот**.

---

## Антипаттерны

| Антипаттерн | Проблема | Что делать вместо этого |
|---|---|---|
| MAS для простой задачи | Накладные расходы на координацию (overhead) больше пользы | Начать с single-agent |
| Сразу 10+ агентов | Почти невозможно отлаживать | Идти постепенно: 2–3 → проверить |
| Нет critic в критичных доменах | Ошибки доходят до пользователя | Добавлять Critic для критичных данных |
| Все агенты на дорогих моделях | Дорого и медленно | Простым агентам — простые модели |
| Нет tracing / observability | Невозможно понять сбой | Логировать каждый TAO-шаг |

### Дополнительные практические замечания
- Мультиагентную систему лучше тестировать **поэтапно**, а не только целиком: отдельно агентов или небольшими группами по 2–3.
- Необязательно использовать **одну и ту же дорогую модель** для всех агентов: тяжелые модели нужны только там, где действительно есть сложная смысловая нагрузка — например, у планировщика или ключевых агентов.

## Мультимодальные агенты

Мультимодальные системы работают не только с текстом.

### Типы мультимодальных агентов

| Тип | Модальность | Примеры задач |
| --- | --- | --- |
| **Зрение (Vision)** | Изображения, видеопоток | Анализ пользовательского интерфейса, OCR документов, визуальные вопросы и ответы |
| **Голос (Voice)** | Аудио | Голосовые ассистенты, колл-центры |

### Дополнительные замечания
- Vision-агенты могут использовать специализированные визуально-языковые модели.
- Voice-агенты обычно строятся как цепочка: **speech-to-text → текстовая бизнес-логика → text-to-speech**.

### Ключевой инженерный вызов voice-агентов
Для естественного голосового взаимодействия задержка обычно должна быть **меньше ~300 мс**, иначе пользователь начинает ощущать паузу как неестественную.

### Мультимодальная мультиагентная система (MAS)
Пример pipeline:
- **VisionAgent**: видит форму бронирования и поля;
- **TextAgent**: заполняет поля на основе запроса;
- **VoiceAgent**: подтверждает действие с пользователем голосом.

### Главная идея
Мультимодальность — это следующий шаг после текстовых агентов:
агенты начинают работать не только с текстом, но и с изображениями, голосом и видео.

## Ключевые выводы

### 1. TAO-цикл — фундаментальный паттерн
Любой агент в основе проходит цикл:
- мысль (Thought)
- действие (Action)
- наблюдение (Observation)

### 2. ReAct лучше CoT для задач с инструментами
Если агент работает с внешними инструментами, ReAct практически полезнее и реалистичнее.

### 3. Мультиагентные системы решают задачи специализации, верификации и масштабирования
Когда один агент перестает справляться, мультиагентная система становится естественным следующим шагом.

### 4. Нет «лучшей» архитектуры
Есть только архитектура, которая лучше подходит под конкретную задачу.

### 5. Мультимодальность — следующий рубеж
Будущие агентные системы выходят за пределы текста и начинают работать с:
- зрением,
- голосом,
- видео
